LA CARTELERA DE HYPE TECNOLOGICO
EXPLICACION GENERAL DEL PROYECTO

1. OBJETIVO DEL PROYECTO

Este proyecto toma un mock de datos que simula la respuesta cruda de la API de YouTube y la transforma en una experiencia mas limpia y util.

La idea principal es separar responsabilidades:

- El backend se encarga de leer, filtrar, transformar y calcular.
- El frontend se encarga de consumir esos datos ya procesados y mostrarlos de forma visual.

El usuario final no ve el payload enorme y desordenado de YouTube.
Solo ve la informacion importante:

- miniatura
- titulo
- autor
- tiempo relativo de publicacion
- nivel de hype


2. ESTRUCTURA GENERAL

El proyecto esta dividido en dos partes:

- backend/
- frontend/

backend:
Aqui vive la API construida con NestJS.
Se encarga de exponer el endpoint /api/videos.

frontend:
Aqui vive la interfaz construida con React + Vite.
Consume el endpoint del backend y renderiza la cartelera.


3. COMO FUNCIONA EL BACKEND

3.1. PUNTO DE ENTRADA

Archivo:
backend/src/main.ts

Este archivo arranca la aplicacion NestJS.
Tambien habilita CORS para permitir que el frontend haga peticiones al backend.


3.2. MODULO PRINCIPAL

Archivo:
backend/src/app.module.ts

Este modulo carga el VideosModule, que es donde esta encapsulada la funcionalidad de videos.

La idea de usar un modulo separado es mantener el codigo escalable y organizado.
Si mañana agregas usuarios, autenticacion o categorias, cada cosa puede vivir en su propio modulo.


3.3. CONTROLADOR PRINCIPAL

Archivo:
backend/src/app.controller.ts

Este controlador tiene rutas simples de entrada:

- GET /
- GET /api

No devuelve videos.
Solo sirve como respuesta base para comprobar que la API esta activa y para indicar cual es el endpoint principal.


3.4. MODULO DE VIDEOS

Archivo:
backend/src/videos/videos.module.ts

Este modulo agrupa todo lo relacionado con videos:

- controller
- service
- repository
- mapper

Esto aplica una idea importante de clean code:
agrupar por feature y no por tipo tecnico global.

En vez de tener todos los controllers juntos y todos los services juntos en carpetas enormes, aqui la feature de videos esta contenida en un solo lugar.


3.5. CONTROLLER DE VIDEOS

Archivo:
backend/src/videos/videos.controller.ts

Aqui se define la ruta:

- GET /api/videos

Este archivo no contiene logica de negocio pesada.
Su responsabilidad es muy pequeña:

- recibir la peticion
- llamar al service
- devolver la respuesta

Eso es intencional.
Un controller limpio no deberia ponerse a calcular hype ni a parsear JSON.


3.6. SERVICE DE VIDEOS

Archivo:
backend/src/videos/videos.service.ts

El service coordina el flujo.

Su trabajo es:

1. pedir los datos crudos al repository
2. transformar cada video con el mapper
3. devolver una lista limpia al controller

Aqui ya se empieza a notar la separacion de responsabilidades:

- repository lee datos
- mapper transforma datos
- service orquesta


3.7. REPOSITORY DE VIDEOS

Archivo:
backend/src/videos/videos.repository.ts

Este archivo se encarga de leer el mock:

- backend/mock-youtube-api.json

Por que existe este repository si solo leemos un archivo?

Porque conceptualmente representa el origen de datos.
Hoy es un archivo local.
Mañana podria ser:

- una API externa real
- una base de datos
- un cache

Entonces el resto del sistema no depende de donde vienen los datos.
Solo sabe que el repository entrega videos crudos.


3.8. INTERFACES DEL PAYLOAD CRUDO

Archivo:
backend/src/videos/interfaces/raw-youtube-response.interface.ts

Aqui se tipa la estructura del JSON crudo.

Esto sirve para varias cosas:

- entender mejor la forma de los datos
- evitar errores
- hacer el mapper mas claro
- dejar documentado que partes del payload externo nos interesan

Se modelan cosas como:

- snippet
- statistics
- thumbnails
- items


3.9. DTO DE RESPUESTA

Archivo:
backend/src/videos/dto/video-response.dto.ts

DTO significa Data Transfer Object.

En este caso define la forma exacta de la respuesta que el backend envia al frontend.

Campos:

- id
- thumbnail
- title
- author
- publishedRelative
- hypeLevel

Esto es importante porque el frontend ya no depende del formato crudo de YouTube.
Depende de un contrato propio y limpio.


3.10. MAPPER DE VIDEOS

Archivo:
backend/src/videos/videos.mapper.ts

Este archivo transforma un video crudo en un VideoResponseDto.

Aqui ocurre lo mas importante de la logica de negocio:

- seleccion de campos utiles
- transformacion de fecha
- calculo del hype
- defaults para datos faltantes

La ventaja de tener un mapper separado es que la transformacion esta centralizada.
Si mañana cambia el payload de entrada o el formato de salida, sabes exactamente donde tocar.


3.11. CALCULO DEL HYPE

La formula base es:

(likes + comments) / views

Reglas aplicadas:

- si el titulo contiene "tutorial" sin importar mayusculas o minusculas, el hype se multiplica por 2
- si no existe commentCount, el hype es 0
- si views es 0, el hype es 0 para evitar division invalida

Ademas el valor final se redondea a 4 decimales.


3.12. TRANSFORMACION DE FECHA RELATIVA

Archivo:
backend/src/common/utils/relative-time.util.ts

Este utilitario convierte una fecha ISO en un texto como:

- Hace 2 meses
- Hace 5 dias
- En 3 meses

Importante:
No usa librerias externas como moment o date-fns.
Todo esta hecho con Javascript nativo.

La utilidad:

- recibe una fecha
- calcula la diferencia contra la fecha actual
- decide si esta en minutos, horas, dias, meses o años
- devuelve un string amigable


3.13. PRUEBAS DEL BACKEND

Archivos:

- backend/src/app.controller.spec.ts
- backend/src/videos/videos.service.spec.ts
- backend/test/app.e2e-spec.ts

Tipos de prueba:

- unitarias:
  verifican piezas concretas de forma aislada

- e2e:
  verifican que la API responda por sus rutas

Las pruebas ayudan a asegurar que la reestructuracion no rompa el comportamiento esperado.


4. COMO FUNCIONA EL FRONTEND

4.1. PUNTO PRINCIPAL DE LA APP

Archivo:
frontend/src/App.jsx

Este componente hace casi toda la orquestacion visual.

Se encarga de:

- pedir datos al backend usando fetch
- manejar loading
- manejar error
- identificar el video con mayor hype
- renderizar la joya de la corona
- renderizar el resto en grilla


4.2. CARGA DE DATOS

Dentro de App.jsx se usa useEffect para disparar la peticion cuando el componente se monta.

La peticion va a:

- /api/videos

En desarrollo, Vite redirige esa ruta al backend gracias al proxy configurado en:

- frontend/vite.config.js

Eso evita tener que escribir la URL completa del backend en el frontend.


4.3. ESTADOS DE INTERFAZ

El frontend maneja tres estados principales:

- loading
- error
- success

loading:
se muestra mientras la peticion sigue en progreso

error:
se muestra si el fetch falla o si la respuesta HTTP no es correcta

success:
se renderizan los videos

Esto mejora mucho la experiencia del usuario porque evita pantallas vacias o comportamientos confusos.


4.4. JOYA DE LA CORONA

Una vez llegan los videos, se usa reduce para encontrar el de mayor hypeLevel.

Ese video se renderiza aparte, en un bloque destacado.

Luego el resto de videos se renderizan en una grilla excluyendo al ganador.

La idea visual es:

- un bloque principal para el video mas importante
- una grilla secundaria para el resto

Eso respeta el requerimiento del reto.


4.5. ESTILOS

Archivos:

- frontend/src/App.css
- frontend/src/index.css

La interfaz se rediseño para que se vea mas profesional y sobria:

- fondo plano
- colores solidos
- tarjetas limpias
- contraste claro
- bloque principal oscuro para destacar la joya

No se usan degradados en la version final.


5. FLUJO COMPLETO DE DATOS

El flujo completo funciona asi:

1. el repository lee mock-youtube-api.json
2. el service pide esos datos crudos
3. el mapper limpia y transforma cada video
4. el controller devuelve el array final en /api/videos
5. React consume esa ruta
6. el frontend detecta el mayor hype
7. la interfaz lo muestra como Joya de la Corona


6. POR QUE ESTA ESTRUCTURA ES MEJOR

La estructura actual es mejor que una version rapida o desordenada por varias razones:

- cada archivo tiene una responsabilidad concreta
- el codigo es mas facil de leer
- el sistema es mas facil de probar
- el proyecto puede crecer sin volverse caotico
- el frontend depende de un contrato limpio
- la logica de negocio no esta mezclada con la logica visual

En clean code esto es clave:

- nombres claros
- funciones con un solo objetivo
- bajo acoplamiento
- responsabilidades bien separadas


7. DECISIONES TECNICAS IMPORTANTES

- Se uso NestJS para estructurar el backend por modulos y capas.
- Se uso React con Vite para un frontend rapido y simple.
- Se usaron DTOs para definir claramente la salida del backend.
- Se usaron interfaces para describir el payload crudo externo.
- Se separo repository/service/mapper/controller para evitar archivos gigantes.
- La fecha relativa se resolvio con Javascript nativo por restriccion del reto.
- Se mantuvo el endpoint /api/videos como fuente oficial para el frontend.


8. QUE SE PODRIA MEJORAR A FUTURO

Si el proyecto siguiera creciendo, los siguientes pasos razonables serian:

- agregar Swagger para documentar la API
- mover configuraciones a variables de entorno
- agregar mas pruebas unitarias para el mapper y utilitarios
- soportar paginacion o filtros
- cachear resultados si el origen de datos fuera externo
- agregar validaciones mas estrictas en DTOs
- extraer componentes visuales del frontend en archivos separados


9. RESUMEN FINAL

El backend actua como un colador inteligente:
lee datos crudos, calcula hype, transforma fechas y entrega un JSON limpio.

El frontend actua como una cartelera visual:
consume ese JSON, detecta el video mas importante y lo presenta con prioridad visual.

El proyecto quedo dividido de forma clara, mantenible y mas profesional tanto a nivel de arquitectura como de presentacion.
